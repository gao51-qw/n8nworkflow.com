## 周期性会议制度

### 1.1 每日站会 (Daily Standup)

**目的**: 核心团队快速同步进度和阻碍

**时间**: 15分钟  
**频率**: 工作日每天 09:30-09:45 UTC+8  
**参与者**: PM、设计师、前端×2、后端、QA（7人核心团队）  
**形式**: 视频会议或文本同步（Slack/钉钉不足时用视频）  
**记录者**: 轮流由小组成员负责

#### 议程（严格控制3项）

1. **✅ 昨天完成了什么** (5分钟)
   - 每人简述已完成的工作项
   - 格式: `[名字]: 完成了 [Issue/功能名] - [简要说明]`
   - 示例: `前端1: 完成了 FilterSidebar 组件架构设计`

2. **📋 今天计划做什么** (5分钟)
   - 每人说明当天预期完成的工作
   - 格式: `[名字]: 计划完成 [Issue/功能名] - [预期交付物]`
   - 示例: `后端: 计划完成 /api/filter-counts 端点开发`

3. **🚧 是否存在阻碍** (5分钟)
   - 描述任何阻挡进度的问题
   - 包括时间预期，优先级标记（P0/P1/P2）
   - 示例: `P0 阻碍: 设计稿未定稿，影响前端1开始工作`

#### 交付物

- **Slack/钉钉同步信息**（emoji 标记优先级）:
  ```
  ✅ 前端1: 完成 FilterSidebar 架构设计
  📋 前端1: 今天实现筛选逻辑
  🚧 P0: 设计稿未定稿，延迟2天
  ```

#### 特殊情况

| 情况 | 处理 |
|------|------|
| 人员缺席 | 提前通知，Slack 异步提交进度 |
| 会议超时 | 优先完成 P0 阻碍讨论，其他转移到单独沟通 |
| 周末或假期 | 休息，下一个工作日继续 |

### 1.2 周一规划会 (Weekly Planning)

**目的**: 确认周目标，对齐团队预期，识别风险

**时间**: 45分钟  
**频率**: 每周一 14:00-14:45 UTC+8  
**参与者**: 全体团队（9-10人：PM、设计、前端×2、后端、QA、内容编辑、可选管理层）  
**形式**: 视频会议，屏幕共享 GitHub Projects Board  
**记录者**: PM

#### 议程分解

**Part 1: 上周完成度评估 (10分钟)**
- 计算完成度百分比: (已完成 Issue / 计划 Issue) × 100%
- 与预期对比（参考项目里程碑）
- 分析延期原因（如果完成度 < 90%）
  - 是否有技术阻碍？是否低估了工作量？是否有意外情况？
- 更新 GitHub Projects labels：`week-1-done`, `week-1-delayed`, `week-1-blocked`

**Part 2: 本周目标确认 (15分钟)**
- 展示本周计划的所有 Issue（应该在上周末已作为 PR 或 Draft 添加）
- PM 说明本周目标和关键里程碑
- 按功能/角色分配任务，明确所有者
- 获得技术认可：是否有技术风险？是否有依赖冲突？
- 目标示例: "P0 功能代码完成 60%，设计稿完成 100%，内容补充 30%"

**Part 3: 资源/依赖风险识别 (15分钟)**
- 识别本周阻碍项
  - 设计稿是否准备好？
  - API 端点准备好了吗？
  - 第三方服务状态？
  - 团队可用性？（有请假吗？）
- 制定风险缓解计划：如果 X 发生，我们的备选方案是什么？
- 更新 GitHub Projects：添加 `blocked` 标签到受阻 Issue

**Part 4: 下周初步预告 (5分钟)**
- 简要说明预期的下周工作
- 提出任何需要提前准备的事项（如设计稿、审批等）

#### 交付物

- **GitHub Projects 文件**:
  - 所有本周 Issue 都应该有 owner（Assignee）
  - 优先级标签已设置 (P0/P1/P2)
  - 估计点数已设置（如使用 Story Points）
  - Done/In Progress/Blocked 状态清晰

- **里程碑进度表** (文本或表格):
  ```
  | 里程碑 | 预期完成周 | 当前进度 | 状态 | 风险 |
  |-------|----------|--------|------|------|
  | M1 需求评审通过 | Week 1 | 100% | ✅ 完成 | - |
  | M2 UI 原型完成 | Week 2 | 80% | 🟨 延期 | 设计稿复杂度 |
  | M3 P0 代码完成 | Week 4 | 45% | 🟩 进行中 | 无 |
  ```

### 1.3 双周设计评审 (Bi-weekly Design Review)

**目的**: 确保设计方案可实现，获得技术反馈，记录改进项

**时间**: 60分钟  
**频率**: 每隔周一次（通常 Week 2, 4, 6, 8）  
**时间点**: 周二 10:00-11:00 UTC+8（在周一规划会之后）  
**参与者**: PM、设计师、前端×2、后端（5人）  
**形式**: 视频会议，设计师屏幕共享 Figma  
**记录者**: PM

#### 议程分解

**Part 1: 新设计稿展示与 Q&A (25分钟)**
- 设计师展示本周期完成的所有设计
  - FilterSidebar 组件设计
  - 工作流详情页布局
  - 响应式设计方案
  - 交互动画演示（Figma prototypes）
- 团队提问和讨论
- 记录反馈（直接在 Figma 中添加评论）

**Part 2: 技术实现可行性评估 (20分钟)**
- 前端评估：
  - 是否能用现有技术栈实现？(Astro + Tailwind + React)
  - 性能影响？(bundle size, render time)
  - 响应式设计的实现方式？
  - 动画实现的复杂度？
- 后端评估：
  - 是否需要新的 API 端点或数据库字段？
  - 数据加载性能？
  - 第三方集成影响？

**Part 3: 反馈收集与修改计划 (10分钟)**
- PM 汇总所有反馈
- 分类反馈：立即修改 vs 后续优化
- 设计师确认修改计划和时间表
- 记录修改任务到 GitHub Issues

**Part 4: 设计评审记录 (5分钟)**
- 确认设计规范已更新
- 更新 REDESIGN_SPECIFICATIONS.md（如有新增）
- 标记 Figma 原型版本号

#### 交付物

- **设计评审记录**: Notion 或 GitHub Wiki 中记录
  ```
  日期: 2026-01-14
  参与者: [名字列表]
  设计稿: FilterSidebar v2.3
  反馈: 
  - [反馈1] → 修改责任人: 设计师 → 完成时间: 2026-01-15
  - [反馈2] → 修改责任人: PM/设计师 → 完成时间: 2026-01-16
  决议: 批准进入开发
  ```

- **技术反馈**: 记录在 GitHub Issue 中
  - 标签: `design-review`, 关联的设计 Issue
  - 描述: 技术团队的可行性评估结果

- **改进清单**: 追踪表
  - 优先级排序（立即 vs 后续）
  - 所有者和完成时间
  - 验收标准

### 1.4 双周技术评审 (Bi-weekly Tech Review)

**目的**: 审查代码质量，评估性能指标，确保测试覆盖，识别部署风险

**时间**: 60分钟  
**频率**: 每隔周一次（通常 Week 2, 4, 6, 8）  
**时间点**: 周三 14:00-15:00 UTC+8（在双周设计评审之后）  
**参与者**: PM、前端×2、后端、QA（5人）  
**形式**: 视频会议，屏幕共享代码仓库和仪表板  
**记录者**: QA 或后端负责人

#### 议程分解

**Part 1: 功能代码实现演示 (15分钟)**
- 开发者演示本周期已 merge 到 main 的功能
  - 实际功能演示（在 staging 或本地）
  - GitHub 代码diff 讲解（关键实现逻辑）
  - 示例: FilterSidebar 的筛选逻辑、缓存策略、URL 同步

- 交互检查：
  - 是否符合设计规范？
  - 是否有 UI/UX 问题？

**Part 2: 性能指标评估 (15分钟)**
- 共享性能基准数据（Lighthouse, Web Vitals）:
  - FCP, LCP, TTI 指标与目标对比
  - 是否有性能下降的迹象？
  - 缓存效率？加载速度？
- 代码质量数据：
  - Test coverage: 达到 ≥80%？
  - Lint 错误数: 应为 0
  - Cyclomatic complexity: 是否有高复杂度函数？
- 浏览器兼容性测试结果

**Part 3: 测试覆盖率、缺陷情况 (15分钟)**
- QA 汇报：
  - 单元测试覆盖率（目标：≥80%）
  - 集成测试通过率（应为 100%）
  - 当前打开的 Bug 数量（按优先级分类）
  - 高优缺陷是否按时解决？
- 缺陷趋势分析：
  - 本周新增 Bug 数 vs 本周解决 Bug 数
  - 缺陷类型分布（前端 vs 后端 vs 集成）

**Part 4: 集成/部署风险评估 (10分钟)**
- 后端或基础设施负责人评估：
  - 数据库迁移: 是否需要 downtime？
  - API 集成: 第三方服务状态？
  - 部署前置条件: 是否都满足？
- Cloudflare 部署检查：
  - 分支预览是否工作正常？
  - 生产部署流程是否清晰？
  - 回滚计划是否准备好？
- 识别任何阻碍项

#### 交付物

- **代码审批**: GitHub Pull Requests
  - 所有 PR 都应该有 ≥1 的审核批准
  - Code owner 应该已 review
  - CI/CD 应该已通过

- **性能报告**: Notion 或 GitHub Wiki
  ```
  日期: 2026-01-17
  FCP: 1.5s (目标: <1.8s) ✅
  LCP: 2.2s (目标: <2.5s) ✅
  TTI: 3.2s (目标: <3.5s) ✅
  Test Coverage: 82% (目标: ≥80%) ✅
  Critical Errors: 0 ✅
  ```

- **测试报告**: 测试人员生成
  - 功能测试通过率: XX%
  - 打开的 Bug 列表 (P0/P1/P2)
  - 测试进度 vs 计划

- **部署 Gate 检查清单**: 发布前使用

### 1.5 周五周总结会 (Weekly Retrospective)

**目的**: 回顾本周成果，识别改进机会，为下周做准备

**时间**: 45分钟  
**频率**: 每周五 17:30-18:15 UTC+8  
**参与者**: 全体团队（9-10人）  
**形式**: 轻松的视频会议，可以考虑虚拟 Retro 工具  
**记录者**: PM

#### 议程分解

**Part 1: 本周完成度回顾 (10分钟)**
- 计算最终完成度：(完成 Issue / 计划 Issue) × 100%
- 相对于计划的表现：
  - 是否达到了周一设定的目标？
  - 有什么超出预期的成就？
  - 哪些没有按计划完成？
- 更新项目里程碑（如已达成某个 Milestone）

**Part 2: 做得好的地方（保持）(10分钟)**
- 团队成员轮流分享本周的亮点
  - 技术上的聪明解决方案
  - 团队协作的例子
  - 文档写得特别好的例子
- 格式: "我欣赏 [人名] 在 [事情] 上的表现，因为 [原因]"
- 记录这些，作为团队文化的一部分

**Part 3: 需要改进的地方（Action Items）(15分钟)**
- 使用 Start/Stop/Continue 框架：
  - **Start**: 我们应该开始做什么？
  - **Stop**: 我们应该停止做什么？
  - **Continue**: 我们应该继续做什么？
- 示例:
  - Start: 早上提交 Daily Standup 不用等到会议
  - Stop: 在周会期间处理突发的 Bug 报告（分散注意力）
  - Continue: 周二的设计评审很有效，保持这个节奏
- 每个改进项应该有：所有者、完成时间、验收标准

**Part 4: 下周风险预判 (5分钟)**
- PM 或技术负责人简述下周的风险点
  - 是否有人员请假？
  - 是否有技术依赖的不确定性？
  - 是否有依赖外部团队的工作？
- 提出预防措施

**Part 5: 鼓励和休息 (5分钟)**
- 团队组长（PM）感谢每个人的贡献
- 宣布下周休息时间（如有）
- 轻松结束会议

#### 交付物

- **周总结文档**: Notion 或 GitHub Wiki
  ```
  周: Week 1 (2026-01-06 ~ 2026-01-10)
  
  完成度: 95% (19/20 Issue 完成)
  
  做得好的地方:
  - PM 的清晰沟通（保持）
  - 前端2 在性能优化上的工作（一流）
  - 设计师的快速反馈循环（保持）
  
  改进项:
  - [ ] Start: 每日 Standup 用异步 Slack 替代（所有者: 无，即刻生效）
  - [ ] Stop: 在周会期间处理 Bug（所有者: PM，生效日期: 下周一）
  - [ ] Continue: 双周设计评审（效果很好）
  
  下周风险:
  - 设计师周三请假，预计影响 2 天
  - 后端需要等待 Mailchimp API 集成确认，预计 2 天
  
  关键决策:
  - 暂时跳过 Newsletter 功能测试，优先完成 FilterSidebar
  ```

- **改进 Action Items 追踪表**:
  - 转移到 GitHub Issues 中
  - 标签: `process-improvement`
  - 优先级和完成时间已设置

## 问题上报与处理流程

### 2.1 优先级定义与响应时间

| 优先级 | 定义 | 响应时间 | 解决时间 | 处理方式 |
|-------|------|--------|--------|---------|
| **P0** | 无法推进，项目阻塞 | 15 分钟 | 30 分钟或升级 | 即时@PM，可能需要紧急电话 |
| **P1** | 影响效率，多个任务受影响 | 2 小时 | 24 小时 | 周会或 Slack 同步，次日解决 |
| **P2** | 长期隐患，单个任务受影响 | 24 小时 | 待定（周会排序） | 记录在 Backlog，周会讨论 |

### 2.2 P0 阻碍处理流程

**触发条件**:
- 开发无法继续
- 测试被完全阻止
- 部署无法进行
- 生产环境宕机

**处理步骤**:

1. **立即上报** (0-5分钟)
   - Slack channel: `#改版-P0-阻碍` (创建紧急 channel)
   - Tag: `@PM @技术负责人 @受影响人员`
   - 格式:
     ```
     🚨 P0 BLOCKED: [阻碍名称]
     受阻者: [名字]
     具体描述: [发生了什么，为什么无法推进]
     影响范围: [哪些功能/人员受影响]
     建议方案: [你认为的解决方案]
     ```

2. **PM 响应** (15分钟内)
   - 确认问题性质
   - 如果可以立即解决：指定 owner 和时间
   - 如果需要协作：召集相关人员进行 5 分钟快速电话
   - 更新 Slack 进度

3. **尝试解决** (15-30分钟)
   - 指定的 owner 尽力解决
   - 每 5 分钟在 Slack 上更新进度
   - 如果 30 分钟内无法解决→自动升级（见下一步）

4. **升级** (>30分钟无解)
   - PM 升级到技术负责人或管理层
   - 可能的升级决策：
     - 临时回滚代码
     - 切换备选方案
     - 调整项目计划
   - 记录升级决策和原因

5. **事后处理**
   - 问题解决 1 小时内，安排 15 分钟复盘会
   - 讨论如何避免重复发生
   - 更新文档或流程

### 2.3 P1 阻碍处理流程

**触发条件**:
- 任务延迟 ≥1 天
- 需要其他人/团队支持但未响应
- 技术决策卡壳（≥2 个观点不一致）

**处理步骤**:

1. **报告** (在 Slack 或周会上)
   - Channel: `#改版-项目`
   - 格式:
     ```
     ⚠️  P1 BLOCKED: [阻碍名称]
     受阻者: [名字]
     开始阻塞时间: [什么时间开始的]
     具体描述: [发生了什么]
     需要帮助: [需要谁或什么资源]
     预期解决时间: [什么时候能解决]
     ```

2. **PM 或技术负责人处理**
   - 在 GitHub Issues 中创建追踪 ticket
   - Tag: `blocked`, `p1-priority`
   - 指定 owner 和完成时间（通常 24 小时内）
   - 在周会期间轮流检查状态

3. **跟进**
   - 如果超过 24 小时仍未解决→升级为 P0
   - 考虑人员调整或范围更改

### 2.4 P2 阻碍处理流程

**触发条件**:
- 低优先级功能受阻
- 长期技术债
- 已知但不紧急的 Bug

**处理步骤**:

1. **记录** 
   - GitHub Issues 中创建 ticket
   - Label: `p2-priority`, `technical-debt`
   - 不需要立即处理

2. **周会排序**
   - 在周一规划会或周五总结会中讨论
   - PM 根据业务价值判断优先级
   - 可能转为 P1 或添加到未来 sprint

3. **追踪**
   - 记录在 GitHub Projects 的 Backlog column
   - 定期（每月）审视，识别是否应该升级

## 代码提交与工作流

### 3.1 Git 分支策略

**主分支**:
- `main`: 生产分支，每个 commit 都经过完整测试和审查
- `staging`: 测试分支（可选），用于灰度前的最后验证

**功能分支**:
- 命名规则: `feature/改版-[功能名]-[简要描述]`
- 示例: 
  - `feature/改版-filter-sidebar`
  - `feature/改版-newsletter-subscribe`
  - `feature/改版-workflow-detail-page-enhancement`

**Bug 修复分支**:
- 命名规则: `bugfix/改版-[功能]-[简要说明]`
- 示例: `bugfix/改版-filter-sidebar-memory-leak`

**规则**:
- 以 `main` 为基础创建功能分支
- 功能完成后创建 Pull Request (PR)
- 至少 1 人审核后才能 merge
- Merge 后删除分支，保持仓库整洁

### 3.2 Pull Request 要求

**PR 创建前的自检清单**:
- [ ] 代码已在本地通过测试
- [ ] 没有 console.error 或 warning
- [ ] 遵循项目代码风格（Prettier/Eslint 已通过）
- [ ] 已添加必要的代码注释
- [ ] 已更新相关文档（API.md, COMPONENTS.md 等）

**Code Review 检查清单**:

Reviewer 应该检查以下项目（参考 QUALITY_ASSURANCE.md 的详细标准）：

- ✅ **功能完整性**: 是否实现了 Issue 中的所有需求？
- ✅ **代码质量**: 
  - 逻辑是否清晰？
  - 是否有明显的 bug？
  - 错误处理是否完善？
- ✅ **性能**: 
  - 有没有不必要的重新渲染？
  - 有没有 N+1 查询？
  - 缓存策略是否合理？
- ✅ **测试**: 
  - 是否有单元测试？
  - 测试覆盖率是否足够？
- ✅ **文档**: 
  - API 文档是否更新？
  - 组件文档是否更新？
  - 有没有复杂逻辑需要注释？
- ✅ **安全**: 
  - 是否有输入验证？
  - 是否有权限检查？
  - 有没有敏感信息泄露？
- ✅ **浏览器兼容性**: 
  - 是否测试了 Chrome/Firefox/Safari？
  - 是否有 console errors?

**反馈方式建议**:
- 🟢 **赞同**: "看起来不错，LGTM（Looks Good To Me）"
- 🟡 **建议**: "考虑用 ... 会不会更好？" 或 "这里的逻辑是 ... 吗？"
- 🔴 **必须修改**: "这是一个 bug，必须修改" 或 "安全影响，需要修改"

### 3.3 提交频率与粒度

**原则**: 最小化 PR，一个功能点 = 一个 PR

**示例**:
- ✅ 好: FilterSidebar 组件架构设计 (1个 PR)
- ✅ 好: FilterSidebar 复选框交互逻辑 (1个 PR)
- ✅ 好: FilterSidebar 样式和响应式 (1个 PR)
- ❌ 不好: FilterSidebar 完整功能从架构到样式 (太大，难审查)

**每周 PR 目标**: 8-15 个 PR per 发开者（具体根据功能复杂度）

**Merge 节奏**:
- 每天应该有 1-2 个 PR merge 到 main
- 周末可以减少

## 文档更新制度

### 4.1 设计文档维护

**所有者**: 设计师（PM 同步）

**流程**:

| 阶段 | 操作 | 时间 | 地点 |
|------|------|------|------|
| **设计完成** | 设计师在 Figma 中完成设计稿 | 设计评审前 | Figma |
| **评审通过** | 设计师标记版本号（v1.0, v2.0），添加交接备注 | 设计评审会后 | Figma Comments |
| **存档** | 设计师导出为 PDF/图片，上传到 GitHub Wiki 或文档系统 | 评审通过后的同天 | GitHub Wiki / Notion |
| **前端实现** | 前端参考设计稿编码，如有疑问询问设计师 | 边实现边沟通 | Slack @设计师 |
| **验收** | 设计师检查前端实现是否符合设计稿，如有差异记录改进项 | 功能完成后 1 周内 | 前端演示 |

**维护的文档**:
- `REDESIGN_SPECIFICATIONS.md`: 整体设计规范（PM 维护主体，设计师更新元素）
- Figma 项目: 高保真设计稿（保持最新版本）
- GitHub Wiki / Notion: 设计交接文档和变更日志

### 4.2 开发文档维护

**所有者**: 技术负责人（各模块负责人维护）

**三类文档**:

#### A. API 文档 (所有者: 后端)

**触发**: 新增或修改 API 端点

**流程**:
1. PR 中必须更新 `API.md` 文件
2. 记录内容:
   ```markdown
   ## POST /api/filter-counts
   获取筛选选项的实时计数
   
   **请求**:
   ```json
   {
     "timePeriod": "last-7-days",
     "price": null
   }
   ```
   
   **响应** (200 OK):
   ```json
   {
     "timePeriods": {...},
     "prices": {...}
   }
   ```
   
   **更新记录**:
   - v1.0 (2026-01-10): 初始实现
   - v1.1 (2026-01-15): 添加缓存头
   ```
3. Code Review 时检查 API 文档的准确性
4. 发布前确认所有新 API 都已文档化

#### B. 数据库变更 (所有者: 后端)

**触发**: 新增表、修改字段、添加索引

**流程**:
1. 在 `scripts/supabase-migrations.sql` 中添加迁移脚本
2. 每个迁移脚本需要：
   ```sql
   -- Migration: 2026-01-10-001-create-workflow-details
   -- Author: 后端开发者名字
   -- Description: 创建工作流详图表用于存储详细内容
   
   CREATE TABLE workflow_details (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     workflow_id UUID NOT NULL UNIQUE REFERENCES workflows(id),
     ...
   );
   
   -- Rollback:
   DROP TABLE workflow_details;
   ```
3. 同时更新 `ARCHITECTURE.md` 中的数据库设计部分
4. 发布前测试迁移脚本的执行和回滚

#### C. 组件文档 (所有者: 前端)

**触发**: 新增组件、修改组件 API

**流程**:
1. 在 `COMPONENTS.md` 中添加或更新组件说明
2. 记录:
   ```markdown
   ## FilterSidebar Component
   
   **路径**: `src/components/ui/FilterSidebar.astro`
   
   **Props**:
   - `filters`: FilterState 对象
   - `onFilterChange`: 回调函数
   
   **示例**:
   ```astro
   <FilterSidebar filters={state} onFilterChange={handleChange} />
   ```
   
   **更新记录**:
   - v1.0 (2026-01-10): 初始实现
   ```
3. Code Review 检查文档准确性
4. 新组件 PR merge 时必须有对应的 COMPONENTS.md 更新
